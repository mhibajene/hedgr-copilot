# HEDGR RETAIL UI PHASE 3 TO PROTOTYPE GATE NOTE

Status: Paper governance artifact only  
Authority: Non-sequencing, non-implementation; repo-native authority remains binding  
Scope: Governance gate between completed Phase 3 paper refinement and any separately approved prototype-only validation pass  
Last updated: 2026-04-21

---

## 1. Purpose

This artifact governs the transition boundary between the completed Phase 3 paper refinement stack and any later prototype-only header validation pass.

It is a gate note, not a prototype brief.

It is not implementation authorization and must not be interpreted as shipped-route approval.

---

## 2. Authority boundary

- Repo-native authority remains binding (`docs/ops/HEDGR_STATUS.md`, `AGENTS.md`, accepted ADR chain, and active repo governance surfaces).
- No implementation authorization is created by this note.
- No shipped-route interpretation is permitted from this note.
- No authority widening is permitted through local interpretation, downstream summaries, or prototype convenience.
- No structural reopening is admissible absent a governance-grade reason tied to explicit reopen triggers.

---

## 3. Current accepted Phase 3 centerline

The accepted paper centerline is already frozen in `docs/ops/HEDGR_RETAIL_UI_HEADER_REFINEMENT_PHASE_3_DECISION_NOTE.md` and remains in force without re-debate:

- money-first shell retained as the active governed base direction
- balance unmistakably first
- trust calm, human, attached, and subordinate
- actions closer to the money anchor but secondary
- `Recent activity` retained as the body transition label
- Family C visible but subordinate trust-layer expression
- Variant A (tight money-anchor cluster) retained as the current upper-shell centerline

Earn-first / APY-first / yield-led framing remains deprioritized in this lane.

---

## 4. Why a gate note is required now

The paper refinement chain is now complete enough that the primary risk is not idea scarcity; the primary risk is discipline loss between a frozen paper centerline and any future prototype-only validation attempt.

Without a gate note, the lane is exposed to:

- prototype drift into redesign behavior
- implicit implementation assumptions entering a paper-only step
- local edits that bypass the frozen Phase 3 centerline
- renewed exploration of already-settled structural questions
- silent widening of authority through language drift or artifact misuse

---

## 5. What a future prototype-only pass is allowed to test

Any future pass is narrowly bounded to validating whether the accepted centerline reads correctly in situ under realistic prototype conditions.

Allowed test focus:

- balance-first readability and first-impression anchoring
- trust-line calmness, human tone, and attachment quality
- action proximity continuity without over-promotion
- `Recent activity` transition quality from upper shell to body
- upper-shell continuity under realistic screen conditions (density, spacing pressure, common viewport contexts)

The pass is validation-only against the accepted centerline, not invention of a new direction.

---

## 6. What remains explicitly out of scope

The following remain blocked unless a separate governance artifact reauthorizes scope:

- shell redesign
- CTA redesign
- Family C redesign
- earn/APY/yield-led reopening
- broader dashboard exploration
- implementation planning or task sequencing
- shipped-route edits or route-level interpretation
- backend, engine, or accounting-truth implications

---

## 7. Success signals for a prototype-only pass

A bounded prototype-only pass is considered governance-successful only if it confirms the frozen centerline without requiring structural expansion:

- readers/users still anchor on balance first
- trust reads as reassurance, not monitoring narration
- actions feel available but not primary
- upper shell reads as trusted everyday money, not a portfolio tracker or utility wallet
- no new structural additions are required to keep the header coherent

---

## 8. Failure / reopen triggers

Header-direction reopening is justified only if prototype evidence materially shows one or more of the following:

- the header still reads as portfolio-led despite the accepted centerline
- trust language consistently reads as system-monitoring or overly generic
- the action cluster consistently overpowers the money anchor
- the current centerline fails under realistic upper-shell use conditions

The following do not count as valid reopen reasons:

- preference drift
- novelty-seeking
- aesthetic restlessness
- generic requests for "more punch"

---

## 9. Required readout expectations

Any later prototype-only pass must produce a governance readout that documents, at minimum:

- what was tested
- what remained fixed from the accepted paper centerline
- what tensions appeared during bounded validation
- whether the paper centerline held under in-situ reading
- whether any explicit reopen trigger was actually met

Readout language must remain paper-governance-oriented and must not imply build execution, implementation authorization, or shipped-route movement.

---

## 10. Working conclusion

The Phase 3 paper centerline is complete and frozen for governance purposes.

Any next prototype pass, if separately approved, must validate that accepted centerline rather than invent a new one.

This note protects the lane from open-ended re-exploration, implementation creep, CTA creep, Family C re-litigation, and silent authority widening.
