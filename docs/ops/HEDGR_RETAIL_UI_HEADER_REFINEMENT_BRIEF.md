## 4. Why a header refinement brief is needed

The shell is now strong enough that the main gains should come from tighter wording, calmer trust phrasing, and stronger everyday-money framing at the header layer.

This pass is needed to:

- reduce portfolio-coded language
- reduce system-authored or monitoring-heavy tone
- tighten the relationship between money and action
- preserve calmness while improving familiarity
- keep trust visible without making the header technical

This is critique and governance guidance only, not implementation direction.

## 6. In-scope refinement themes

### 6.1 Balance labeling

`Portfolio value` should be pressure-tested against more everyday-money framing.

Candidate pressure tests:

- remove label entirely
- replace with `Total balance`

The goal is less portfolio logic, not less clarity.

### 6.2 Stability phrasing

The current trust line should be simplified toward more human, confidence-building language.

For this pass:

- `Monitoring` should not be a lead primary state
- reassurance should remain visible
- the line must stay descriptive, not promissory

### 6.3 Activity naming

`Recent money state` should be replaced with more familiar money-language framing.

Candidate framings:

- `Recent activity`
- `Recent`
- `Activity`

`Recent activity` is the strongest current centerline candidate.

### 6.4 Money-to-action relationship

The action cluster should read more clearly as an extension of the money anchor.

For this pass:

- actions should sit closer to the balance flow
- they must remain secondary
- this is not a CTA redesign pass

### 6.5 Allocation / trust subordination

The allocation / trust block should remain visible but quieter.

For this pass:

- it should support first impression, not compete with it
- it may be pressure-tested for lower visual prominence
- it remains part of Hedgr truth, not optional decoration

## 8. Working product principles for the pass

### 8.1 Everyday money, not portfolio management

The header should feel like a money account, not a wealth-tracking surface.

### 8.2 Calm reassurance, not system narration

Trust should be communicated through confidence-building phrasing, not monitoring language.

### 8.3 Money first, action second

The balance should anchor the header; actions should extend from it, not rival it.

### 8.4 Trust visible, not technical

Hedgr's differentiation should remain present, but quiet.

### 8.5 Subtraction over invention

The next gains should come from simplification and tightening, not added complexity.
