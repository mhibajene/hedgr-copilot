# HEDGR CODEX TRIAL RUN BRIEF

Status: Governance guidance only  
Authority: Non-sequencing, non-implementation  
Scope: First governed Codex pilot for bounded critique and validation inside the allocation legibility lane  
Last updated: 2026-04-21

---

## 1. Purpose

This document defines the **first governed Codex pilot** inside Hedgr: a real, bounded exercise that tests Codex as a **secondary operator** under existing repo and Project Ops discipline.

The goal is to learn whether Codex improves **critique quality** and **validation depth** without competing with **Cursor** (primary execution), **repo-native authority**, or **founder judgment**.

This trial is **not** a build authorization, sequencing instruction, or approval of Codex-led repo work. It exists to prove **disciplined usefulness**, not agent autonomy.

---

## 2. Authority boundary

This pilot **does not** alter the authority order in `docs/ops/HEDGR_CODEX_USAGE_CONVENTION.md`.

- **Founder** direction and approval remain primary.  
- **Repo-native authority** (`docs/ops/HEDGR_STATUS.md`, accepted ADRs, `AGENTS.md`, doctrine) remains the institutional operating system.  
- **Project Ops / `docs/ops`** remains the governance and traceability layer for bounded artifacts.  
- **Cursor** remains the primary repo execution surface.  
- **Codex** is limited to **bounded verification and testing** for this pilot—no primary building, no direct mutation of repo truth, no product arbitration.

---

## 3. Why this pilot is needed

The Codex **usage convention** is documented; the system still needs **one** concrete, low-risk exercise to validate operating discipline under live use.

The pilot should answer:

- whether Codex improves review quality (signal vs noise)  
- whether Codex surfaces hidden risks, tensions, or strong counterarguments  
- whether the workflow stays **governed** when a second agent is in the loop  
- whether Codex can add validation throughput **without** authority confusion  

That validation is safest in a lane that is **already bounded, documented, and critique-oriented**—not in open-ended product or implementation work.

---

## 4. Chosen pilot lane

The first pilot lane is:

**Allocation legibility** — the **Allocation / Trust Detail** presentation layer on the calm-money **C3** retail dashboard baseline, as framed in the existing allocation exploration stack.

This lane is appropriate because it is:

- **already bounded** — same C3 order, subordinate allocation, no IA widening in the pattern/micro-test frame  
- **already documented** — legibility brief, visual pattern comparison, same-frame micro-test, scorecard, pass resolution  
- **already critique-oriented** — Paper exploration posture; non-sequencing, non-implementation  
- **already non-implementing** — explicit subordination to `HEDGR_STATUS.md` and `AGENTS.md`  
- **high enough value** — trust-surface legibility, precision implication, and performative drift are real product risks  
- **low enough authority risk** — no authorized shipped change emerges from the pilot by default  

**Pilot inputs in this lane** include the artifacts listed in Section 6, which collectively define Patterns **A / B / C**, fixed frame assumptions, and critique criteria.

---

## 5. Pilot mode selection

The first Codex pilot should use **exactly** these modes:

**Execution mode:** `READ_ONLY`

### 5.1 Verifier

**Purpose:** independently review the allocation **pattern comparison** and **micro-test** setup for drift, overclaim, hidden precision implication, or doctrine tension (including consistency with informational allocation posture and ADR **0013** alignment as referenced in the lane docs).

### 5.2 Tester

**Purpose:** run an **independent critique read** against the allocation **A / B / C** pattern set using the **fixed criteria** already established in the lane (including `docs/ops/HEDGR_RETAIL_UI_CRITIQUE_SCORECARD.md` and the rubric implied by the micro-test and pattern comparison).

**Explorer** mode is **not** the primary mode for this first pilot. The goal is **validation**—pressure-testing what is already framed—not generating new option sprawl.

---

## 7. Pilot question set

Ask Codex **critique and validation** questions—not open-ended design generation. Examples:

1. What **hidden risks** exist in Patterns **A / B / C** that the current docs may be underweighting?  
2. Does any pattern imply **more precision, accounting truth, or visual authority** than the current read-only, informational posture can honestly support?  
3. Is the current **working centerline candidate (Pattern C)** actually the strongest under the **stated rubric and scorecard criteria**, or is there a **stronger counterargument** for A or B on doctrine-safe grounds?  
4. Where does the allocation lane risk drifting into **performative**, **chart-like**, or **performance-theater** behavior despite guardrails?  
5. Do the allocation docs contain **internal contradiction**, **duplicated authority claims**, or **silent scope creep** relative to `HEDGR_STATUS.md` and the lane’s own authority boundaries?  

These are **review prompts**, not mandates to redesign the screen or rewrite repo truth.

---

## 8. Expected Codex output type

The pilot should request:

- **one** bounded **critique readout** (structured findings against A / B / C)  
- **one** **risk-focused challenger pass** (adversarial to overclaim and hidden-mechanics drift)  
- **one** **consistency / drift check** across the allocation docs and the convention’s handoff rules  

Each output should also include a short provenance note indicating whether it relied on: repo authority, bounded lane artifacts, external tools / connected systems, or memory / inferred continuity.

Expected character of outputs:

- **findings**, **tensions**, **counterarguments**, **validation signal**  

Not:

- **approvals**, **implementation instructions**, **shipping decisions**, or **sequencing**  

---

## 9. Output handling rule for the pilot

Codex outputs remain **non-authoritative by default** per `docs/ops/HEDGR_CODEX_USAGE_CONVENTION.md`.

Required handling:

- **Review** outputs through founder / Project Ops / **Cursor** workflow before treating them as inputs to execution.  
- **Capture** any material, durable findings in **repo-native** `docs/ops` (or ADR flow if scope escalates—unlikely for this pilot).  
- **Do not** let Codex **directly rewrite** allocation lane docs or prototype truth without human review and explicit absorption into the governed chain.  
- If Codex detects conflict between repo authority, lane artifacts, external sources, or memory / inferred continuity, it must surface the conflict explicitly and must not reconcile it by inference.

Handoff chain:

**Codex output → review → repo-native doc or Cursor brief → founder / governance review**

No silent codification and no second execution source of truth.

---

## 10. Success criteria

A successful first pilot:

- surfaces **at least one** meaningful risk, tension, or blind spot **not already foregrounded** in the lane (or deepens an existing one with a sharper counterargument)  
- produces **bounded**, **usable** critique—specific to A / B / C and the fixed frame—not generic product advice  
- **does not** create authority confusion (Codex reads as verifier/tester, not decider)  
- keeps the workflow **disciplined** (inputs explicit, scope allocation-only, outputs reviewed before absorption)  
- improves **judgment** for reviewers more than it adds **noise**  

---

## 11. Failure conditions

Treat the pilot as needing **tightening before broader use** if:

- Codex outputs read like **product authority** or **implementation orders**  
- scope **widens** beyond allocation legibility (new IA, new surfaces, repo-wide UX mandates)  
- the review devolves into **generic** or **unmoored** commentary  
- the workflow **bypasses** Project Ops traceability or **founder** judgment  
- outputs **contradict** repo-native authority without flagging tension as a finding (silent override)  
- memory, Chronicle, or inferred continuity is treated as current artifact truth  
- output provenance is unclear or omitted  
- Codex smooths over source conflict instead of surfacing it explicitly  

---

## 12. Suggested pilot outputs

Likely downstream artifacts (all critique-oriented, governance-safe):

1. `HEDGR_CODEX_ALLOCATION_CRITIQUE_READOUT.md` — consolidated independent readout (findings only; non-authoritative until reviewed)  
2. A **short finding summary** folded into an existing allocation lane doc **only after** review and explicit decision to absorb  
3. A **brief recommendation** on whether Codex should advance to a **second pilot lane**—not a mandate to expand  

Naming and placement of absorbed findings remain subordinate to `HEDGR_STATUS.md` and normal `docs/ops` practice.

---

## 13. Working conclusion

The first Codex pilot should be **small**, **bounded**, and judged on whether it **improves governed review**—not on volume of output or speed of iteration.

**The first Codex trial should prove disciplined usefulness before Codex is allowed to widen its role.**
