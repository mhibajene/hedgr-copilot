## 6. Balance labeling comparison

### Option A — no label

- **Read:** The amount is treated as self-evident primary money information.
- **Strengths:**
  - Maximum subtraction; lowers chance of portfolio-coded framing.
  - Feels closer to everyday account behavior.
  - Keeps visual/cognitive emphasis on the number itself.
- **Risks / drift:**
  - Can reduce immediate semantic precision for first-time viewers.
  - May read as stylistic minimalism rather than intentional clarity.
  - In some contexts, ambiguity can invite backfilled portfolio interpretation.
- **Working assessment:** Strong for de-financializing the tone, but only admissible when surrounding copy already anchors obvious money context.

### Option B — `Total balance`

- **Read:** Direct account-style money anchor with explicit label.
- **Strengths:**
  - Clear and familiar everyday-money wording.
  - Avoids portfolio-coded `value` language while preserving explicitness.
  - Supports fast orientation without technical framing.
- **Risks / drift:**
  - Slightly more text weight than no label.
  - Can feel generic if trust phrasing nearby becomes bland.
- **Working assessment:** Current strongest governed centerline. Best balance between clarity and portfolio-language reduction.

**Current working centerline:** Prefer `Total balance`, with no-label retained only as a controlled subtraction fallback if clarity remains intact.

## 7. Trust / stability phrasing comparison

Guardrails for this section:
- confidence-building
- non-technical
- non-promissory
- non-system-authored

Explicitly rejected lead directions:
- monitoring as primary state
- system-heavy narration
- operational/surveillance wording
- empty reassurance that erases Hedgr trust signature

Disciplined candidate set:

### Candidate 1 — `Protected with Hedgr safeguards.`

- **Read:** Calm, human reassurance with explicit Hedgr trust signature.
- **Strengths:**
  - Clear confidence signal without technical narration.
  - Preserves distinct Hedgr posture through `safeguards`.
  - Avoids promise language and avoids surveillance tone.
- **Risks / drift:**
  - `Protected` may be over-read as absolute by some users if unsupported by surrounding trust context.
- **Working assessment:** Strong candidate; keep under discipline to avoid absolute-security interpretation drift.

### Candidate 2 — `Hedgr safeguards help keep your money steady.`

- **Read:** Human and explanatory, emphasizes stability as assistance rather than guarantee.
- **Strengths:**
  - Non-promissory due to `help`.
  - Strong everyday-money tone.
  - Preserves trust distinctness and calmness.
- **Risks / drift:**
  - Longer phrasing; may add copy weight in a compact header.
  - `steady` can read soft if adjacent labels are also softened.
- **Working assessment:** High-quality trust language, but slightly heavier than ideal for upper-shell brevity.

### Candidate 3 — `Built for safer everyday money.`

- **Read:** Broad reassurance tied to daily use.
- **Strengths:**
  - Simple and calm.
  - Avoids monitoring/system tone.
- **Risks / drift:**
  - Too generic; weak Hedgr-specific trust signature.
  - `safer` without explicit trust mechanism can flatten differentiation.
- **Working assessment:** Admissible as fallback, but weaker than candidates that preserve explicit Hedgr safeguards language.

**Strongest current centerline:** `Protected with Hedgr safeguards.`  
**Inadmissible directions:** Monitoring-led statements, operational telemetry phrasing, system-authored narration, and generic reassurance copy that removes Hedgr trust identity.

## 8. Activity naming comparison

Baseline pressure-test set:
- `Recent activity`
- `Recent`
- `Activity`

### Option A — `Recent activity`

- **Read:** Familiar, clear, and broadly understood list label.
- **Strengths:**
  - Highest immediate clarity.
  - Everyday-money convention without technical jargon.
  - Keeps semantic precision while staying compact.
- **Risks / drift:**
  - Slightly conventional; limited brand distinctiveness on its own.
- **Working assessment:** Strongest governed centerline unless contradicted by future evidence.

### Option B — `Recent`

- **Read:** Minimal shorthand relying on contextual inference.
- **Strengths:**
  - Very lightweight.
  - Supports subtraction.
- **Risks / drift:**
  - Ambiguous object of recency (transactions, notices, system events).
  - Can feel stylistic rather than clear.
- **Working assessment:** Too compressed for primary naming; not preferred.

### Option C — `Activity`

- **Read:** Generic activity feed label without time anchoring.
- **Strengths:**
  - Clear noun.
  - Short and stable.
- **Risks / drift:**
  - Weaker immediacy than `Recent activity`.
  - Can feel app-generic and less money-specific.
- **Working assessment:** Serviceable but less precise than `Recent activity`.

**Current working centerline:** `Recent activity`.

---

## 9. Cross-cutting findings

- The strongest copy path is subtraction with explicit money clarity, not novelty phrasing.
- Portfolio-coded drift is reduced most effectively by replacing `value` semantics with `balance` semantics.
- Trust copy performs best when human and calm, with explicit Hedgr safeguards language and no monitoring-first narration.
- Activity naming should prioritize clarity over stylistic compression.
- Header confidence increases when money anchor is explicit, trust is subordinate, and list naming is conventional but precise.

---

## 10. Working recommendations

- Use `Total balance` as the primary balance label centerline for this phase.
- Use `Protected with Hedgr safeguards.` as the trust/stability centerline for current testing language.
- Use `Recent activity` as the activity naming centerline.
- Continue subtraction discipline; avoid adding explanatory/system copy unless clarity materially requires it.
- Keep trust visible and subordinate; do not let trust line become a monitoring or operational narrative.

These are working paper recommendations only, not implementation approval.
