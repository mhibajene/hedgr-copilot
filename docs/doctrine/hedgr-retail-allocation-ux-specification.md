# Hedgr Retail Allocation UX Specification

Version: 1.0
Status: Binding
Scope: Core Retail Interface (Light Mode Canonical)

---

## Purpose

This specification defines how the Hedgr Stability Model™ must be represented within the retail interface.

It translates constitutional doctrine and allocation policy into enforceable UX structure.

This document governs layout hierarchy, information sequencing, visual weighting, and behavioral constraints.

No design iteration may override this specification without governance review.

---

# I. Canonical Design Mode

Light Mode is the canonical reference environment.

- All hierarchy decisions are defined in Light Mode first.
- Dark Mode must preserve hierarchy and risk-return balance.
- No theme may alter allocation emphasis.

Light Mode must communicate:
Clarity. Stability. Transparency. Authority.

---

# II. Dashboard Hierarchy (Non-Negotiable Order)

The retail dashboard must follow this visual sequence:

1. Portfolio Value (Primary Anchor)
2. Stability Status Indicator (Contextual Signal)
3. Allocation Split (Buffer vs Yield)
4. Yield Detail (Secondary)
5. Risk & Exposure Granularity (Expandable Layer)

This hierarchy must not be inverted.

---

# III. Portfolio Value Rules

Portfolio Value is the primary emotional stabilizer.

Requirements:

- Largest numeric element on dashboard.
- No aggressive red/green flashing.
- Daily change not dominant.
- No celebratory animation.
- No urgency framing.

Optional contextual line:
“Managed under the Hedgr Stability Model™.”

Portfolio Value communicates reassurance, not excitement.

---

# IV. Stability Status Indicator

A Stability Indicator must appear beneath Portfolio Value.

Permitted formats:

- Stability: Strong
- Stability: Within Expected Range
- Stability: Protective Mode Active

Prohibited:

- Numeric gamified scores
- Leaderboard-style ranking
- Competitive comparison

Purpose:
Reinforce bounded volatility and structural discipline.

---

# V. Allocation Panel Structure

The allocation interface must display:

1. Stability Buffer (First)
2. Conservative Yield Allocation (Second)

Stability Buffer must visually precede Yield Allocation.

Requirements:

- Buffer % clearly visible.
- Buffer amount clearly visible.
- Yield % visible but secondary.
- No isolated APY without allocation context.
- Risk tier visible alongside Yield.

The Buffer must visually feel foundational.
Yield must feel controlled.

---

# VI. Visual Weighting Constraints

Under no circumstances may:

- APY font size exceed Buffer percentage size.
- Yield color intensity exceed Stability color intensity.
- Yield placement precede Buffer placement.
- “Earn” language replace “Allocate” language.

Language rules:

Use:

- Allocate
- Stability Buffer
- Conservative Yield
- Exposure
- Variance

Avoid:

- Maximize
- Boost
- Unlock
- Beat
- Outperform

---

# VII. Risk Visibility Standard

Risk must never be abstracted away.

The interface must allow access to:

- Exposure distribution
- Protocol diversification
- Buffer coverage logic
- Stress-response explanation

APY may never be displayed without risk context accessible.

---

# VIII. Stress Mode Interface Behavior

When Volatility Response Mechanism activates:

- Yield visual emphasis decreases.
- Buffer visual emphasis increases.
- Stability indicator updates.
- Calm explanatory copy displayed.
- No alarmist UI.
- No flashing alerts.

Example tone:

“Volatility elevated. Your Stability Buffer has been increased to protect principal.”

Stress Mode must feel controlled, not reactive.

---

# IX. Comparison & Gamification Prohibition

Explicitly banned:

- Leaderboards
- Best-performing allocation labels
- Trending strategies
- Streak rewards
- Yield badges
- Countdown timers
- Limited-time yield boosts
- Social proof comparisons

Hedgr does not promote competitive performance culture.

---

# X. Interaction Model

Core Retail Allocation is system-managed.

Users:

- May view allocation structure.
- May view risk breakdown.
- May not override Buffer minimum.
- May not exceed exposure caps.
- May not fully deploy capital.

Manual override features are reserved for future Pro tier and must be visually segmented.

---

# XI. Pro Tier (Future Segmentation Rule)

If Hedgr Pro is introduced:

- It must be visually distinct.
- It must not alter core interface hierarchy.
- It must not bleed into core allocation visuals.
- It must include enhanced risk disclosure.
- It must retain crisis-trigger enforcement.

Pro must feel advanced, not more aggressive.

---

# XII. Design Review Enforcement

Every new retail feature must answer:

1. Does this encourage reactive behavior?
2. Does this elevate yield over stability?
3. Does this reduce visibility of Buffer?
4. Does this introduce urgency psychology?
5. Does this compromise premium-controlled posture?

Any “Yes” requires governance review.

Tier 3 violations are blocked.

---

# XIII. Behavioral Outcome Target

After 12 months of usage, a Hedgr user should:

- Expect capital to be partially buffered.
- Accept lower yield for stability.
- Think in allocation terms.
- Value predictability over peak returns.
- View volatility as managed, not chased.

If the interface promotes any other behavior, it is misaligned.

---

# Closing Principle

The Retail Interface is not a marketing surface.

It is the visible layer of the Hedgr Stability Engine™.

It must communicate discipline,
reinforce predictability,
and encode treasury logic without preaching.

Design serves stability.
Not excitement.
